# Architecture

Copinance OS follows **Clean Architecture (Hexagonal Architecture)** with clear separation of concerns.

## Overview

```
┌─────────────────────────────────────────┐
│           CLI Layer                     │
│  (User Interface)                       │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│      Application Layer                  │
│  (Use Cases - Business Operations)      │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│         Domain Layer                    │
│  (Core Business Logic - No Dependencies)│
│  • Models (Entities, Value Objects)     │
│  • Ports (Interfaces)                   │
│  • Services (Business Logic)            │
│  • Validation                           │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│      Infrastructure Layer               │
│  (Implementations - Adapters)           │
│  • Repositories                         │
│  • Data Providers                       │
│  • Analysis executors                   │
│  • Container (Dependency Injection)     │
└─────────────────────────────────────────┘
```

## Core Vocabulary

Copinance OS centers on **analysis** as the main user-facing concept. The canonical terms are:

| Term | Description | In code |
|------|-------------|---------|
| **Analysis request** | What the user wants to know (target, mode, options). | `AnalyzeInstrumentRequest`, `AnalyzeMarketRequest` (`application.use_cases.analyze`) |
| **Analysis mode** | How the analysis runs: deterministic (fixed pipeline) or question-driven (AI with tools). | `AnalyzeMode` (`auto`, `deterministic`, `question_driven`) |
| **Analysis result** | Outcome of a single run: success, payload, or error. | `RunJobResult` (`success`, `results`, `error_message`) |
| **Job** | Optional orchestration envelope: scope, target, timeframe, and internal execution type. Used for job-based execution (e.g. queues, batch). Not persisted. | `Job` (`domain.models.job`) |
| **Executor** | Internal component that performs one kind of analysis (deterministic instrument, deterministic market, question-driven). Selected by the runner from the job. | `AnalysisExecutor` port; implementations in `infrastructure.executors` |

The public API is **analysis requests and use cases**; **Job** and **AnalysisExecutor** are for library integrators who need custom orchestration or new execution strategies.

**Routing:** Executor selection is based on the job’s `execution_type`. Derive it from scope + analysis mode via `execution_type_from_scope_and_mode(scope, mode)` in `application.use_cases.analyze` when building jobs from requests.

## Layer Responsibilities

### Domain Layer
- **Core business logic** with zero external dependencies
- **Entities**: Job, Stock, AnalysisProfile
- **Ports**: Interfaces for data providers, repositories, job execution (JobRunner, AnalysisExecutor)
- **Services**: Complex business logic
- **Validation**: Domain validation rules

### Application Layer
- **Use cases**: Business operations (CreateProfile, GetInstrument, etc.)
- **Default job runner**: Optional implementation of `JobRunner`; consumers can replace it for custom orchestration (queues, retries, routing)
- **Orchestration**: Coordinates domain objects and infrastructure

### Infrastructure Layer
- **Implementations**: Concrete implementations of domain ports
- **Repositories**: Data persistence (memory, file, database)
- **Data Providers**: External data sources (yfinance, EDGAR, etc.)
- **Analysis executors**: Deterministic and question-driven execution (implement `AnalysisExecutor`)
- **Container**: Dependency injection configuration

### CLI Layer
- **User interface**: Command-line interface
- **Error handling**: User-friendly error messages
- **Output formatting**: Rich terminal output

## Design Patterns

### Repository Pattern
Abstracts data access:
```python
# Port (Domain)
class StockRepository(ABC):
    async def get_by_symbol(self, symbol: str) -> Stock | None

# Implementation (Infrastructure)
class StockRepositoryImpl(StockRepository):
    async def get_by_symbol(self, symbol: str) -> Stock | None:
        # Implementation using storage backend
```

### Job Runner (optional)
Running a job is the consumer’s responsibility; the library provides a default implementation you can replace:
```python
class JobRunner(ABC):
    async def run(self, job: Job, context: dict[str, Any]) -> RunJobResult:
        # Find executor, build context, execute
```

### Dependency Injection
Manages dependencies through container:
```python
# Default job runner (replaceable)
runner = container.job_runner()
result = await runner.run(job, context)
```

## Extension Points

The architecture provides 22 port interfaces for extension:

- **5 Data Provider interfaces**: Market, Alternative, Fundamental, Macro, Base DataProvider
- **1 Analyzer interface**: LLM
- **6 Strategy interfaces**: Screening, Due Diligence, Valuation, Risk, Thematic, Monitoring
- **4 Tool interfaces**: Tool, ToolParameter, ToolSchema, ToolResult
- **2 Repository interfaces**: Stock, AnalysisProfile
- **Job runner**: JobRunner (default implementation provided; replace for custom orchestration)
- **Analysis executor**: AnalysisExecutor (one implementation per analysis mode)
- **2 Storage interfaces**: CacheBackend, Storage

See [Extending](extending) for how to implement these.

## Benefits

- ✅ **Testability**: All layers are easily testable
- ✅ **Flexibility**: Swap implementations without changing business logic
- ✅ **Maintainability**: Clear separation of concerns
- ✅ **Extensibility**: Easy to add new features
