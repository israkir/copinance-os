# Architecture

Copinance OS follows **clean (hexagonal) architecture** with strict boundaries: **domain** contracts, **data** and **infra** adapters, **core** orchestration and pipelines, **AI** for language and tool use only, and **interfaces** (CLI) at the edge.

## Design principles

1. **Financial and quantitative work is deterministic** — Implemented in domain models, data providers, analytics (e.g. QuantLib), and `core.execution_engine` executors. Outputs are structured and testable.
2. **LLMs do not replace computation** — They explain, route questions to tools, and format answers. Tool implementations perform fetches and numeric work.
3. **Contracts over raw blobs** — Prefer Pydantic models and explicit result types between layers; avoid passing untyped pandas frames across module boundaries in new code.
4. **One front door for job execution** — **`ResearchOrchestrator`** wraps the **`JobRunner`** port so workflows and integrations do not scatter direct calls to executors. Override **`JobRunner`** only when you need custom orchestration (retries, queues, multi-tenant routing).

## Package layout (source)

| Area | Role |
|------|------|
| `copinance_os.domain` | Entities and value objects (`Job`, `Stock`, market types, profiles). **Ports**: data providers, repositories, storage, `JobRunner`, `AnalysisExecutor`, tools, analytics estimators, `LLMAnalyzer`. Strategy protocols for future screening / valuation flows. |
| `copinance_os.research.workflows` | Application-facing use cases and request types (`AnalyzeInstrumentRequest`, market search, profile CRUD, backtest workflow inputs). |
| `copinance_os.core.orchestrator` | **`ResearchOrchestrator`**: `run_job`, and workflow helpers (e.g. simple backtest entry). Default analyze runners delegate here. |
| `copinance_os.core.execution_engine` | Executors: deterministic instrument/market analysis, question-driven analysis. |
| `copinance_os.core.pipeline.tools` | Tool implementations (market data, fundamentals, regime, macro) and registry. |
| `copinance_os.data` | **Providers** (yfinance, FRED; **SEC/EDGAR** in `data.providers.sec` via edgartools), **cache**, **repositories**, **loaders**, **analytics** (e.g. `data.analytics.options` for BSM Greeks). |
| `copinance_os.ai.llm` | LLM provider adapters, configuration, prompt resources. |
| `copinance_os.infra.di` | Dependency injection: `get_container()`, lazy container proxy, singletons. |
| `copinance_os.interfaces.cli` | CLI entry (`copinance` script). |

## Vocabulary

| Term | Meaning | Typical code location |
|------|---------|------------------------|
| **Analysis request** | Typed input for progressive analyze flows (`AnalyzeInstrumentRequest`, `AnalyzeMarketRequest`). | `copinance_os.research.workflows.analyze` |
| **Analysis mode** | `auto`, `deterministic`, or `question_driven`; resolves how a job is built. | Same module + `resolve_analyze_mode` |
| **Job** | Non-persisted execution envelope: scope, symbols, timeframe, **`execution_type`**. | `copinance_os.domain.models.job` |
| **Execution type** | String discriminator for which executor runs (e.g. `deterministic_instrument_analysis`). Derive with `execution_type_from_scope_and_mode`. | `research.workflows.analyze` |
| **ResearchOrchestrator** | Facade over **`JobRunner.run`**: the supported entry for running a constructed `Job` with context. | `copinance_os.core.orchestrator.research_orchestrator` |
| **AnalysisExecutor** | Implements one analysis style; validates and executes a `Job`. | `copinance_os.core.execution_engine` |
| **Tool** | Callable unit exposed to the LLM with schema and structured results. | `copinance_os.core.pipeline.tools`, `domain.ports.tools` |

## Orchestration: what to call

- **Recommended:** `container.research_orchestrator()` then `await orchestrator.run_job(job, context)`. This matches how default analyze runners work.
- **Advanced:** `container.job_runner()` returns the same **`JobRunner`** port **`ResearchOrchestrator`** uses; override it for custom global behavior (e.g. middleware, alternate routing).

## Dependency injection and startup

The global container is a **lazy proxy**: nothing heavy runs at import time. Resolving `get_container()` or attributes like `container.research_orchestrator()` builds the real container once; many services are **singletons** per process.

See [Using as a Library](/getting-started/library#configuration) for `get_container(...)` options (LLM, FRED, cache, storage, prompts).

## Extension points (ports)

Implementations live in `data`, `ai`, and `infra`; contracts live under **`copinance_os.domain.ports`** and related domain modules:

- **Data:** `MarketDataProvider`, `FundamentalDataProvider`, `MacroeconomicDataProvider`, `AlternativeDataProvider`
- **Analytics:** `OptionsChainGreeksEstimator` (default QuantLib BSM path)
- **AI:** `LLMAnalyzer`
- **Execution:** `JobRunner`, `AnalysisExecutor`
- **Persistence:** `StockRepository`, `AnalysisProfileRepository`, `Storage`, cache backends
- **Tools:** `Tool`, `ToolSchema`, `ToolParameter`
- **Strategies:** screening, due diligence, valuation, risk, thematic, monitoring (protocols for future workflows)

Step-by-step: [Extending](extending). Interface listing: [API Reference](/api-reference/data-providers) and `src/copinance_os/domain/ports/`.

## Benefits

- **Testability** — Ports are easy to fake; executors and tools are isolated.
- **Replaceability** — Swap providers or LLM backends without changing domain rules.
- **Clarity** — Numbers and series come from data and analytics layers; prose and routing from AI and CLI.
